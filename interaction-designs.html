Redesign User Admin

Task Description:

I got a task to redesign the Admin Manager of Amadeus Social Media Suite ( ASMS: A Management System for Travel Industry Companies to manage their social channels ).

This Admin Manager is where all registered users display.The user structure in ASMS is complicated, it has 5 layers: Market -> Chain -> Travel Agency -> Account -> Travel Agent . Administrator should be able to create/edit/delete/search/monitor user, as well its parents.

The old design of Admin Manager didn’t show this structure clearly, and some functionality were hidden. There were a lot of usability issues, and it was simply ugly. A brand new design is required.

Quick Links:

  • Data Structure Map
  • Who-Wants-What Graph
  • Solution Expectation
  • Wireframes

Challenge:

Navigate users easily between the 5 layers structure, and provide search functionality. No matter what user want to do (search/ create/ edit), they cannot be stucked.

Method/Skill Used:

Problem Solving Design Pattern Wireframe Sketching HTML, CSS, Javascript

Before Design -1: Reduce Problems

Explanation

After a long meeting with product manager, the first thing I did was to ask to reduce the structure complexity. I gave product manager two suggestions:

Suggestion 1. Merge Market and Chain. The reason is: Market and Chain are 90% similar and for most markets there is only 1 chain for all users.

Suggestion 2. Hide Accounts from UI. Because Travel Agent and Account are same from Administrator/user’s point of view. The concept behind it is: You can login with “One Account”, and then you can switch between different “Travel Agents” which belongs to Different “Travel Agencies”. So when create a new “Agent”
= 1. create a new “Agent” and a new “Account” +assign it to a “Travel Agency”
= 2. select existing “Agent” + create a new Account to this “Agent”, assign this Account of the “Agent” to a “Travel Agency” Why not just delete this “Account” layer from surface (it is still in database, just not visible for end-user anymore). In this case Administrator don’t need to think anymore: Should I create an Account or an Agent? what’s the difference again?

Finally, suggestion 2 was taken. Suggestion 1 was refused, because market and chains are two different marketing concepts, both are needed for sales team. And we decide to make one more small change, use general names instead of travel industry terminology. To sum up, the finally structure look like this:
Market -> Group -> Company -> User
After decision was made, I took a step forward, I make a structure map, combined the required functions and the 4-layer structure into one map, which gave me a clear view of the data-structure and main functionality.

Explanation

For me, design always equals to solving problems, users’ needs are the problems, providing what they want is to solve the problems. So here comes to the problem-identifying graph:
Who are my target users? what are their problems?

During Design - 1: wireframe Sketching

Explanation

Landing Page:
From the “who-wants-what” problem identifying graph, I knew that, System Administrator wants 3 things: create market/group/company/user, search any of them and edit any of them.
So for the landing page, I decided to give system administrator all functions in a straightforward way. By one look, administrator understand that they can create any entity they want, or search for anything they want to view or edit.

(note: for monitoring users, because it was not a major function, it has been postponed. )

Search Functionality:
From the landing page, it is simple, users know what they want to search and then they directly go to the desired search field. But for displaying search result, it is tricky, how to show the search result in a structured way? how could it be easy for users to start a new search in the search result page? Searching with the 4-layer structure is not as easy as Google search, displaying everything in a list as Google does, won’t help administrators filter out what they want, and give them a overview the huge and complex user-system they are managing. So we have to show the 4-layer structure somehow in the result page.
There are two ways to do that:
1. Structure search results
2. Structure search queries
I made two sketch based on those two proposals. And then I imaged some scenarios which System Admin might encounter with. And then I applied them with the two solutions. I found out that structuring search queries was easier for making a new search and it had a cleaner design.

Verify the low-fi design:

Redesign Digital Management System

Task Description:

Digital Management System is a part of Amadeus Social Media Suite where user could manage images, PDF files, and youtube videos. Within the system user can either edit the files or simply share them to the social sites which are connected with the system.

Same as all old components in Amadeus Social Media Suite, design was not optimised for users, some functionalities are hidden, working processes are not smooth. There are died points, unclear steps and miss leading instructions etc. A brand new, simplified modern design is needed.

Quick Links:

  • Functional bits
  • Wireframes
  • HTML prototype
  • Polishing/Refining works

Challenge:

There are a lot functionalities in this component, and image itself is colorful. So providing a balance between a clean view and a full functionalities access becomes the biggest challenge for me.

A 3-weeks-design-time is given to produce an interactive HTML click dummy. It is expected to be ready to deliver to development team. And also HTML, CSS, and javascript(partially) are expected to be production code.

Method/Skill Used:

Sketching Wireframe HTML, CSS, Javascript

Time Spent:

3 weeks: around 150 hours

Before Design: Functional Bits

Explanation

The functional list I got from Product Manager was really long, and it took me quite a while to find them out one by one in old system. But it worth spending time to dig up all those functions and then target main design problems there.

The biggest problem I found out in the old design is that all the media types (photo, docs, PDFs, youtube videos) are mixed in one list view without having any signifier which tells user who they are. And it became worse, because different type has different functionalities. For exmaple, due to technical limitations, the system doesn't support direct video uploading from users' computer, users can only Synchronize their Youtube Channels in order to have videos in system. And in the old design, there is no way users would find out this information. What they could do is repeatedly trying to upload videos and then got some error messages which they don't understand, finally became angry. Those hidden-functionalities problems are quite common in the old design. So the first thing I did before designing is to restructure the functions, and group them in a more user friendly way. Depends on file types, and depends on function types, separate them into different groups/UI-groups.

I imagined myself as user, give me several task I want to do. For example, upload a photo, edit a photo, synchronize a youtube video, create a collection. In this way, I broke functionalities into several groups. Each group has a different UI for users to accomplish some similar tasks.

Last but not least, I used a red pen to market out what were the problems in old design for each functionality, to remind myself to avoid them in the new design.

During Design - phase 1: wireframe Sketching

Explanation

Sketching is my favorite part during design process. Any flash idea can be captured by just drawing lines on a paper. Usually, I draw fast, only-readable by myself wireframes at the beginning stage. I draw at least 5 different ideas, no matter they are good or not. And then analyse them, select one or two which I like most, redo the wireframe again. After that I would spend hours looking at them:

Imagine what I would do if I was a user, how could I do it. Is there any places that I will get stuck? Is there buttons/links/icons I need to think before I click?

After this step. I will finalize my wireframes. If there are still more than one options, for example, the overview of Digital Management System (See the image below), I will write down advantages and disadvantages of each single design to help me making the finally decision.

During Design - phase 2: HTML Click dummy

Explanation

From my opinion, after having an idea or an image how the design should look like, creating HTML click-dummy usually goes quite smoothly. Of course you will find out that something in the wireframe which do not fit with real data and you need to do some adjustments, but the main direction won't change much in most of cases.

The biggest issue I had in this design task is that there were too many things you could do with photos and too much info of photos you should show to users, plus photos themselves are colorful objects If I put everything on the surface like my original idea in the wirefames, it looks overwhelmed. That is also the main reason why I abandoned the card view before. But even for the list view, which has much more space, it looked crowded when I made most functionalities visible together with photos. I really want to give users an impression that, they can have a rest here, enjoy looking at beautiful images, take a pause from boring daily works. So finally, I decided to use a hover effect to hide most functions, and give user a clean view when they land on this page.

After Design: Polishing/refining after testing

Explanation

After finished the design, I gave the click-dummy to a small group of people who use the old digital system daily, and let them play with it freely. The feedback was good. Not many changes during this phase were needed, only some small things were pointed out, for example, the color of the heart icon and the check icon were not bright enough. When a image had too many red/green colors, they were not obvious enough to identify. So I increased the contrast between the background grey and icons colors.

Some bigger changes came after the development. With real data, performance were not as good as it showed in click-dummy. Sometimes, it is easier to improve it with better design than spending weeks kicking off the technical bottle nick For example, "marking as favorite" button, in click-dummy, users see feedback immediately, but in real world it usually takes 2 to 4 seconds to get response from server. So during these 2 to 4 seconds, users doesn't know what is happening. And they might click again and again on the icon, which of course causes troubles. To avoid the confusion, I add an small animation to the icon. After clicking, the icon will spin slowly until we get the response from server. Animation clearly tells users that the icon is already being clicked. At the same time, it brings fun. I found out that some users did enjoy clicking on it again and again just to see the animation.

Before Design: User Testing

Explanation

There are several similar games in the market already. So before design I made a little research, asked people to played those games and watched their behaviors. New Design: Notification Center

Task Description:

This new product idea came from a customer complain. In ASMS, it had a simple but super annoying notification design. Each single time users login in, they will receive a big pop-up window, which shows all unread notifications. Of course you can close popup easily by clicking on close button, which is what most users do. But it repeatedly show up every 2 hours if you don't “dismiss” the messages. The only way to dismiss it is to click on the gear mean and then choose “ mark as read ” for each single message you’ve received.

After month and month, you might get a list of 100 messages which show up every 2 hours. If I was the customer, I would kill the page immediately and blame the system of stopping me from working.

Challenge:

Due to some organization issues, I was the Product Owner of Notification Center, and then take over the design task after a while. So keeping the balance of caring about functions and paying more attention to designs becomes the biggest challenge for this task.

Method/Skill Used:

Functional Bits Sketching Wireframe HTML, CSS, Javascript

Before Design: Targeting Problems

Explanation

As a general method I usually use to design a new product. I asked myself 3 questions. Whom am I design this product for ? What’s their problems? What functionality I can provide to solve their Problems?

Here came this base document which I used as an instruction for design.


There are four main issues:
1. Number of notifications grows too fast
2. Too many error messages
3. Notifications cannot be easily dismissed
4. User preferences are different

For the first problem, it is not really a design issue. I solved it by changing most notification into flash message alert, because there are a lot of notifications which are just feedback info after some immediate action.

For the second one, it is both functional issue and design issue. What is definition of error message? How to display the error message, so that it could get user’s attention, at the same time avoid being annoying. In our case, we degraded some error messages to warning.

The third one is definitely a design problem. Mark as read should be the easiest task ever, one click, it’s done.

For the last one, we need a new functional improvement, which allows user to freely choose which kind of notification they want to receive, and in what kind of way, notification inbox, pop-up, or email ?

After analysed these problems, I made another list: expected result/functions. And then mapped them with the problems, to make sure that all problems are covered.

New Design: Pre-booking Ticket

Task Description:

IBE, so called International Booking Engine is widely used by travel agencies across Europe. The goal is to make the booking process more social for travellers, to allow them to share booking information easily and by that drive more traffic to the travel agency.

The task was to create a responsible web page to display basic information about a flight booking. The page should show the price of the flight at the time of sharing and allow user to check the current price.

Quick Links:

  • Goal-oriented design
  • Low-fi design
  • HTML Mockup

Challenge:

We want travellers to share and visit the page as much as possible, so the page must have a modern and attractive design. But the information we get from the booking engine is hardly rich: departure and destination airports, price, flight time - that’s all. How to use so little data to create a rich experience was the biggest challenge.

Method/Skill Used:

Goal-oriented design Sketching Wireframe HTML, CSS, Javascript

Before Design: Goal-oriented analysis

Explanation

In order to encourage users to share the page, we need to provide what they want. I asked myself: why would a user want to share a booking information? In my opinion , there are three main scenarios, or use-cases, as explained in the storytelling graph below:

1. It is a group trip, information needs to be shared among all travellers. In this case, the time of travel and price of the flight are the critical data.

2. User wants more friends to join the trip, so a welcoming photo of the destination, to be shared again by friends of friends, is good idea. Of course, price and travel dates are important as well.

3. User doesn't know where to go, so she/he just shares something to get inspiration. For this scenario, a nice photo and a place where friends can make comments and discuss the trip become more important.

Before Design: Sketching

Explanation

As a first step of trying to satisfy all three requirements, I made a sketch - see the picture below. In short, I wanted to show who has shared this pre-ticket and to display all the basic information in a central place. The background should be a nice image of the travel destination. A facebook comment box and several share buttons should be available in the footer.

After having this first idea, I needed to do some research. First, where could I get background photos? Second, is it legal to store facebook user data. Unfortunately the answers for both questions were no: it is too expensive to buy images and the image quality cannot be guaranteed if we use images available online randomly, and there might be legal issues.

Because of that I had to make another design. I simplified the page to make it look more like a ticket, while adding some decorations around it to make sure it doesn't look boring at the same time. Then, basing on this, I created a design for the mobile version too.

The next step was actual HTML page creation.
